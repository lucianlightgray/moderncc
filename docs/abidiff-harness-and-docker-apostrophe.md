`tools/abidiff-docker.sh` (added 2026-07-22) is a mcc↔gcc mixed-object ABI differential: it compiles a self-contained no-headers corpus (structs by-value/return, HFA, varargs, bitfields, unions, long double) with BOTH mcc and gcc, links every cross mix (mcc-lib+gcc-main / gcc-lib+mcc-main / both-mcc), and requires the exit code to match the all-gcc reference. Wired as `abi-diff-{arm64,riscv64,arm,amd64}-docker` ctests. It finds a bug class that self-host and `extlink` CANNOT: **caller/callee-asymmetric non-conformant ABI** — mcc compiles both sides consistently so every self-test passes, but crossing to gcc exposes it. Three real bugs found so far, all documented in docs/TODO with disasm + repro: (1) x86_64 mixed INTEGER+SSE small-struct (`classify_x86_64_merge` collapses to one whole-aggregate class) — OPEN, `-DABI_SKIP_MIXED`-guarded on amd64, needs per-eightbyte reclassification; (2) riscv64 caller FP-arg + indirect struct return — FIXED 2026-07-22 (`d9f3942e`): `gfunc_call` advanced its `sa` param-walk for the prepended hidden sret arg (arg 0), pushing the trailing arg off the named list → riscv64 unnamed-FP-in-GP rule sent it to a GP reg; fix = don't advance `sa` for the sret arg (return-type struct size>16); (3) riscv64 FP-arg spill to GP — FIXED 2026-07-22: LP64D passes exhausted FP args (>8) in GP arg regs then stack, but mcc sent them straight to the stack. Clean fix WITHOUT fmv: mcc already places variadic doubles in GP regs (reg_pass named=0 → INT, emission integer-loads the bits), so re-classify a scalar FP arg as INT when its FP regs are exhausted but a GP arg reg is free, in BOTH gfunc_call (caller) and func prolog (callee); the arg flows the existing GP path. Scalar-only, gated on exhaustion → ≤8-FP code byte-identical. Each guarded case reproduces its bug (remove `ABI_SKIP_*`). (4) over-aligned struct-by-value register rounding — FIXED 2026-07-22 (arm64 + armv7): mcc applied the AAPCS register even-rounding to `__attribute__((aligned(16)))` structs, but gcc/clang round only for NATURAL alignment. arm64 fix = `arm64_natural_align16(CType*)` gates `arm64_pcs_aux` (~1032); caller+callee share it (one gate). armv7 fix = `arm_pcs_natural_align(CType*)` (max natural member align, clamp [4,8]) gates the rounding in BOTH `assign_regs` (caller) AND `gfunc_prolog` (callee ~1328) — separate fns on armv7, both needed. Only when `align>8` so ≤8 code byte-identical; stack-slot alignment still honors the attribute. Validated: abidiff arm64+armv7 green, native macOS ctest 5511/5511, arm-inline 83911/0. NOTE: on disasm mcc was CONSISTENT (both sides rounded), not asymmetric — trust disasm over inferring caller-vs-callee from mix pass/fail when both-mcc passes. The differential's mcc-lib+gcc-main / gcc-lib+mcc-main / both-mcc split pinpoints caller-vs-callee (both-mcc passing = self-consistent-but-non-conformant). METHOD NOTE: when a probe diverges from gcc, check clang too before calling it a bug — if gcc≠clang it is an impl-defined corner, not an mcc bug; here gcc==clang confirmed it. The differential found 4 real bugs across register-classification (riscv64 sret [fixed], riscv64 FP-spill, x86_64 mixed-SSE) AND layout/alignment (arm64/armv7 over-align) dimensions — probing register-EXHAUSTION and alignment boundaries is where the remaining bugs cluster. NOTE: qemu-riscv64 differentials (promote/inline/abidiff) can segfault under heavy session load (many nested-qemu containers exhaust the Docker VM) — pristine HEAD segfaults identically, so isolate a suspected regression by re-running the same test on stashed HEAD, and trust `IDENT_OK`/byte-identity over a flaky qemu run.

**Second differential — DWARF/gdb** (`tools/dwarfgdb-docker.sh`, ctest `dwarf-gdb-docker`): builds mcc, compiles with `-gdwarf`, links with gcc, runs gdb, and asserts every backtrace frame has a source line. Found + FIXED (2026-07-22) a general `.debug_line` bug: mcc emitted `DW_LNE_end_sequence` at the last line-CHANGE address, not the code end (the line PC only advances on a line change, mccdbg.c ~1384), so the LAST function in a CU got a zero-length final line range and gdb showed it as `main ()` with no `at file:line`. Fix = advance PC to `text_section->data_offset` before end_sequence in `mcc_debug_end`. gdb needs real ptrace so this only runs host-native (arm64 on Apple Silicon), NOT under qemu-user; the line table is arch-independent so one native arch validates the fix for all. A SECOND, more serious bug (2026-07-22) FIXED via the same harness: `break FUNC` broke in the WRONG function for SINGLE-LINE functions (decl line == first-statement line). mcc emits `DW_LNS_set_prologue_end` but it only flags the next row, and the next `mcc_debug_line` is suppressed on same-line (mccdbg.c ~1386) → no post-prologue row → gdb prologue-skip overshoots to the next function (e.g. `break usept` stopped in `addup` with garbage args — the apparent "-O1 garbage locals" were this overshoot, not a location bug). Fix = a `dwarf_line.prologue_end` flag that bypasses the same-line suppression once. Lesson: differentials against an EXTERNAL consumer (gcc linker, gdb) find latent bugs self-host masks — extend to new consumers/dimensions, not just more ABI cases. When gdb reads "garbage" locals, FIRST confirm the breakpoint is in the right function (decode DWARF low_pc vs objdump addresses + line table) before blaming variable locations.

**Third differential — C++ unwinder / .eh_frame CFI** (`tools/ehunwind-docker.sh`, ctests `eh-unwind-{arm64,riscv64,amd64}-docker`): a g++ main throws a C++ exception from a callback invoked by an mcc-compiled C bridge; the catch only fires if libgcc `_Unwind` can walk mcc's `.eh_frame` for the bridge frame (vs all-g++ reference). VALIDATED CONFORMANT 2026-07-22 on arm64 (native) + riscv64 (cross g++ + qemu) — no bug; the RUNTIME CFI check extlink (link-only) never did. armv7 SKIPs: ARM EABI uses EHABI (`.ARM.exidx`) not `.eh_frame`, and C default has no unwind tables in BOTH mcc and gcc (both terminate identically → mcc matches gcc, no divergence). Three external-consumer differentials now exist (linker=extlink, gcc=abidiff, gdb=dwarfgdb+g++-unwind=ehunwind); their clean-bug veins are largely mined (recent probes across ~17 ABI dims + stepping/locals/unwind all conformant). Then a CROSS-OBJECT TLS case in abidiff found + FIXED (2026-07-22) a 5th bug: mcc compiled an extern `__thread` REFERENCE to an `STT_NOTYPE` undef symbol, not `STT_TLS`, so GNU ld rejected linking it against a TLS definition ("bad value" / "TLS definition mismatches non-TLS reference"). ROOT CAUSE (took 3 iterations of tracing — ruled out update_storage, greloca, set_elf_sym first): the symbol IS created STT_TLS in put_extern_sym2, then `mccelf_end_file` (mccelf.c:190) deliberately downgrades EVERY undefined GLOBAL to STT_NOTYPE for object output (MCC_OUTPUT_OBJ). Harmless for regular symbols (GNU ld tolerates NOTYPE undef) but wrong for TLS. FIX = exclude STT_TLS from that reset (one line). LESSON: when a symbol has the right type at creation but the wrong type in the object, the culprit is the object-write finalizer (mccelf_end_file), not the codegen/reloc path. Session tally: 7 bugs FIXED (4 non-x86 ABI + 2 DWARF + 1 TLS/ELF), plus tracked-open x86_64 mixed-SSE and x86_64 packed (unverified).

**8th bug — riscv64 `_Complex` arg spilled to STACK not GP regs when FP regs exhausted — FIXED 2026-07-22** (found by a NEW `_Complex` ABI differential). Same flavor as bug #3 (scalar FP-spill) but for a 2-FP-member aggregate. RISC-V passes a `float`/`double _Complex` (== struct of 2 FP members) in FP regs only if BOTH fit; else it reverts to the INTEGER convention (`double _Complex` 16B → 2 GP regs a_n,a_n+1; `float _Complex` 8B → 1 GP reg packed; split/stack if GP also full). mcc's `gfunc_call` (~642) / `gfunc_prolog` (~934) sent an FP-insufficient 2-FP aggregate STRAIGHT TO THE STACK (`nregs==2 && prc[1]==prc[2]==FLOAT && areg[FP]>=limit`), so `double _Complex cd_after7(double a..g, double _Complex z)` (a-g fill fa0-fa6, z needs 2 FP regs, only fa7 free) had mcc stack z while gcc/clang use a0,a1. Both cross-mixes failed, both-mcc passed (self-consistent-non-conformant); gcc==clang confirmed. FIX (reuses existing machinery, mirrors #3): when a 2-FP aggregate hits the FP-insufficient threshold (`areg[1]>=15` caller / `>=7` callee), re-invoke `reg_pass(type,prc,fieldofs,/*named=*/0)` — the `!named` path already classifies by SIZE into the integer convention (8B→1 GP, 16B→2 GP, integer-load fieldofs), so it flows through the existing GP-reg/split/stack path like a variadic value. Handles float (1 reg) AND double (2 reg) complex + any 2-double struct past FP exhaustion. Gated on exhaustion → ≤6-FP code byte-identical. riscv64-gen.c only; arm64/armv7/x86_64 already conformant. Guard: `cx_cf_id`/`cx_cd_id`/`cx_cd_after7`/`cx_cf_after7` added to abidiff corpus, HELPER-FREE (pass/return/add only). GOTCHAS learned building the `_Complex` differential: (a) `__int128` is NOT an mcc source type (VT_QLONG is internal-codegen-only, no `TOK___int128`) — can't differential-test it; (b) mcc lowers `_Complex` MUL/DIV to its OWN runtime helpers `__mcc_cmul`/`cmulf`/`cmull` (`runtime/lib/complex.c`), like `__va_arg` — an mcc object doing complex mul/div needs the mcc runtime linked (gcc uses libgcc `__muldc3`, auto-linked); keep differential corpus mul/div-free to avoid the dep; (c) a standalone ARM (armv7) differential MUST build mcc with the FULL flag set `-DMCC_TARGET_ARM=1 -DMCC_ARM_VFP=1 -DMCC_ARM_EABI=1 -DMCC_ARM_HARDFLOAT=1` (like the abidiff arm cell) + link `-static` — with only `-DMCC_TARGET_ARM=1` mcc emits NO `.ARM.attributes` (EABI version 0) and GNU ld rejects linking against gcc-hardfloat EABI-v5 objects ("EABI version 0 but target has version 5"). That is a probe-config error, NOT an mcc bug. Session tally now 8 bugs FIXED (5 non-x86 ABI + 2 DWARF + 1 TLS/ELF).

**Docker-script apostrophe gotcha (hit twice):** the container body of every `tools/*-docker.sh` runs inside `docker run ... bash -c '…'` — a SINGLE-quoted string. Any apostrophe inside (even in a C comment in an embedded heredoc, e.g. "mcc's") terminates the quote and breaks parsing (symptoms: `here-document ... delimited by end-of-file`, `<word>: command not found`). Write "the mcc X" / "mcc" not "mcc's". `bash -n` does NOT catch it (the script is syntactically valid; it fails at container runtime). Verify with: `awk 'NR>=<block-start> && /'\''/' tools/<x>-docker.sh`. See [[docker-macos-tmp-mount]] (mounts only share /Users — keep workdirs under the repo, never scratchpad).
