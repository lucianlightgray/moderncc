Update 2026-06-30 (full local+cross+qemu-docker run, all-debug/diagnostics): green everywhere after 5 macOS-specific fixes. (1) `-Wcomment`: a literal `+/-/*` in two comments (src/mccgen.c:2405, tests/cli/cases.h) → reworded `+, -, or *`. (2) cli `int64_c_type` failed because `runtime/include/stdint.h` chose the `INT64_C`/`UINT64_C` suffix from `__SIZEOF_LONG__==8` alone → `L` on macOS, but mcc's `__INT64_TYPE__` is `long long` on Darwin/BSD LP64 (only LP64 *Linux* is `long`, per runtime/include/mccdefs.h). Fixed to `#if __SIZEOF_LONG__==8 && defined __linux__`, which exactly mirrors mccdefs.h; `__linux__` is emitted only for the default ELF target (mccpp.c predef table), so Apple/BSD correctly get `LL`. (3) cli `ident_backslash_no_hang` hard-coded `timeout 10` which doesn't exist on macOS → switched to the runner's existing `{TIMEOUT}` placeholder (expands empty when no coreutils timeout). (4) **diff3-suite**: on macOS `find_program(gcc)`+`find_program(clang)` both resolve to Apple clang, so the three-way differential degenerates — every intentional/impl-defined divergence (cleanup, bitfields_ms, predefined_macros, c11_freestanding) became a *false* mcc-divergence FAIL (the gcc!=clang "impl-defined" escape can't fire). Added `same_compiler()` (compares `--version`) → SKIP (rc 77) with reason when the two refs are identical; exec golden suite still covers those programs. Also added a dependency-free hang-guard (`timeout`/`gtimeout`/POSIX-shell-watchdog fallback) around BOTH the build and run steps in tests/diff3/runner.c — `tests/exec/features_c99_c11/cleanup.c` (a goto+`__cleanup__` torture test) can spin Apple clang's `-cc1` codegen for minutes AND its compiled binary loops at runtime, which previously hung the whole suite >90min with no ctest timeout firing. (5) cross build only: `-Wpointer-sign` at src/mccgen.c:9296-9297 in the `write_ldouble` branch gated `(__aarch64__||__riscv) && target x86` (only compiled by the i386/x86_64 cross compiler on an arm64 host) — `read64le/read16le` take `unsigned char*`; cast `(char*)s`→`(unsigned char*)s`. qemu-docker matrix = 22/22 (all archs×glibc/musl + arm64-osx macho row); diagnostics+cross ctest 30/30 (passes+legit skips).

Key non-obvious fact: on Mach-O/arm64 mcc sets `MCC_USING_DOUBLE_FOR_LDOUBLE` (src/mcc.h), so `long double` is carried as `VT_DOUBLE | VT_LONG` (the `VT_LONG` bit keeps it a *distinct type* from plain `double` for `_Generic`). `arm64-gen.c` still `#define`s `LDOUBLE_SIZE 16`, but the parser remap in `mccgen.c` (~line 5101) wins, so `sizeof(long double)==8`. Anything keyed off `VT_BTYPE` alone (e.g. the complex-type cache in `mk_complex_type`) must also account for `VT_LONG`, or `double _Complex` and `long double _Complex` collapse.

Mach-O TLS is now implemented (2026-06-29) via native TLV descriptors: `src/objfmt/mccmacho.c` synthesises a `__DATA,__thread_vars` section of 24-byte `{thunk,key,offset}` descriptors (`macho_tls_setup`/`macho_tls_finalize`), maps `.tdata`/`.tbss`→`__thread_data`/`__thread_bss`, repoints each STT_TLS sym at its descriptor, binds the thunk to `__tlv_bootstrap`, and sets `MH_HAS_TLV_DESCRIPTORS`; codegen in `arm64-gen.c` (`arm64_macho_tls_addr`) and `x86_64-gen.c` (`gen_macho_tls_base`) calls the descriptor thunk to get the per-thread address (both guarded `#ifdef MCC_TARGET_MACHO`). Gotcha found & fixed: the front end tagged the rodata literal that spills an FP constant `VT_TLS` — `gv` in `mccgen.c` now strips it (was latent on ELF/PE too), and x86_64 `gen_opf` force-loads a TLS float lvalue instead of using it as a memory operand (SSE-prefix/REX ordering). `tls`/`tls_aggr` goldens now run on Darwin (removed `os!=Darwin`). Verified arm64 native + x86_64-osx via Rosetta (`arch -x86_64`).

C11 `<threads.h>` now ships as `runtime/include/threads.h` — a Darwin shim over pthread using the `__has_include_next` delegate-or-define pattern (glibc/musl get the system header via `#include_next`; only macOS uses the shim). `c11_threads` golden runs on Darwin now (still `os!=WIN32`).

Update 2026-07-04 (commit 9c48e53d — fix the macOS regression from b2ae4912 "Rename runtime lib to mccrt, embed it"). That refactor turned `MCC_EMBED_RTLIB` ON by default, which **forces the native mccrt to be host-CC built** (see `mcc_build_rtlib`, the `if(native AND (USE_HOSTCC OR EMBED_RTLIB))` gate) to break the embed build-cycle. On Darwin the host CC emits **Mach-O** objects, but **mcc's object loader reads only ELF** (`mcc_object_type` in mccelf.c recognizes ELF magic + `ar` only; there is NO macho relocatable-object reader — only `macho_load_dll`/`macho_load_tbd` for .dylib/.tbd). So mcc couldn't link the host-CC runtime: `runmain.o: unrecognized file type` + `unresolved reference to ___builtin_bswap16/___atomic_add_fetch_*/___mcc_cmul` → 38 ctest fails incl. hello-run. **Fix = fall back from embed to the sidecar libmccrt.a on Darwin** (3-line CMake guard mirroring the existing WIN32 one), because with EMBED off + USE_HOSTCC off the runtime is built by **mcc itself → ELF → loadable** (exactly what the pre-refactor parent e461a3e1 did: `runmain.o` built by `mcc`, not gcc). Key non-obvious fact: **mcc emits ELF `.o` even when targeting Mach-O** (its internal interchange format) yet still defines `__APPLE__`, `__MCC__`, `__leading_underscore`, and `__USER_LABEL_PREFIX__=_` — so the ELF it writes carries **leading-underscore** symbol names (Mach-O ABI) and standard ELF `.type/.size/.L`. The embed build ALSO surfaced 3 host-CC Mach-O compile errors in the runtime .c (only hit when gcc/clang compiles them, i.e. USE_HOSTCC or the old embed path): (a) `__attribute__((alias))` needs a **weak** alias on Mach-O and **gcc-16 ICEs on `__attribute__((weak,alias))`** (`assemble_alias` at varasm.cc:6679) → emit the alias via assembler `.set` with a leading `_` from `__USER_LABEL_PREFIX__` (chained `.set` resolve fine; builtin.c/stdatomic.c); (b) the Mach-O assembler has **no `.type`/`.size`** directives; (c) it needs **assembler-local labels to start with `L`, not `.L`** ("conditional branch requires assembler-local label … is external"). Gate (b)/(c) and the alias `.set` behind **`__APPLE__ && !__MCC__`** so mcc keeps its ELF spelling (proven path) and only a foreign Mach-O compiler gets the workaround. macos + macos-cross presets (gcc-16): clean build, **803/803 ctest**; clang host build warning-free after clearing a pre-existing `-Wcomment` in mcc.h (`arch/*-gen.c` → `arch/<target>-gen.c`). Longer-term alternative to the sidecar fallback would be a real macho relocatable-object reader in mcc (then host-CC embed would work), or a stage0-mcc bootstrap to build+embed an mcc-compiled runtime.

Remaining macOS items are non-defects: the kernel-fused libSystem path (libmalloc/locale-stdio/dyld) needs `MCC_DARWIN_HOST=ON` and has no runnable test behind it; the `diff3` divergences are non-macOS-specific (cleanup intentional, bitfields_ms impl-defined, predefined_macros is the default `__STDC_VERSION__`, c11_freestanding is bundled-vs-system headers).
