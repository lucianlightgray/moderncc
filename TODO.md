# C11 Conformance TODO

Goal: implement **every detail** of ISO/IEC 9899:2011 (C11), tracked here and
backed by regression tests across all targets (x86_64, i386, ARM, AArch64,
RISC-V 64), platforms (ELF/PE/Mach-O), and both glibc and musl, until every
feature is implemented and every test passes.

Reference: **N1570** (the C11 working draft; clause numbers below cite it).
Each item records: the standard requirement, the current `mcc` status (with how
it was verified — `file:line` or a test result), and the precise gap.

Default standard in `mcc` is **C99** (`cversion=199901`, `src/libmcc.c:856`);
C11 is selected with `-std=c11`. Many items below were verified at `-std=c11`.

Severity tags:
- **[BUG]** — wrong codegen, crash, or rejects-valid C11 (silent miscompile or hard failure).
- **[FEATURE]** — a C11 feature not implemented at all.
- **[DIAG]** — a required constraint violation that is accepted with no diagnostic (C11 requires "a diagnostic"); or a diagnostic that is wrong/misleading.
- **[OPT]** — optional in C11 (a conforming implementation may omit it); listed for the "implement everything" goal.

Legend: `[ ]` not started · `[~]` partial · `[x]` done+tested.

## 1. Lexical elements & translation phases (§5.1.1.2, §5.2.1.1, §6.4)

### Keywords (§6.4.1)
- [x] **[FEATURE] §6.4.1 `_Imaginary` keyword** — DONE: added `TOK_IMAGINARY` (`src/mcctok.h`) and a `parse_btype` case that errors "imaginary types are not supported" (matching clang's exact wording; gcc also rejects). `_Imaginary` is now a proper reserved keyword rather than an undeclared identifier; `_Complex` is unaffected. Imaginary-type *arithmetic* (Annex G) is intentionally not implemented (neither gcc nor clang implements it, yet both still define `__STDC_IEC_559_COMPLEX__` — mcc matches). Fires on all 5 arches. Tested via cli `imaginary_not_supported`.

### Identifiers & universal character names (§6.4.2, §6.4.3)
- [x] **[FEATURE] §6.4.2.1 UCN escape form in identifiers** — DONE: `\uXXXX`/`\UXXXXXXXX` are now accepted as identifier characters, both at the start of and within an identifier. Added `decode_ucn()` (`src/mccpp.c`) and intercepted `\u`/`\U` in the tokenizer's leading-`\` case and in the identifier slow-path loop, decoding the code point and appending its UTF-8 via the existing `cstr_u8cat` — so a UCN names exactly the same identifier as the literal UTF-8 spelling (`int é=10; é+=1;` works). Line splices (`\`+newline in an identifier) and the genuine "stray '\'" diagnostic are preserved; UCNs in string/char literals are unaffected. Verified vs gcc+clang (all three accept; `\u`/`\U`/mixed forms run identically); compiles on all 5 arches. Tested via exec `ucn_identifiers`. (A UCN straddling an 8KB buffer-refill boundary falls back to the stray path — vanishingly rare; documented in `decode_ucn`.)
- [x] **[DIAG] §6.4.3p2 UCN value < 00A0 not rejected (in identifiers)** — DONE for the case that matters: a UCN **in an identifier** that designates a code point < 00A0 (except 0024 `$`, 0040 `@`, 0060 `` ` ``) or in the surrogate range D800–DFFF now errors "universal character \uXXXX is not valid in an identifier" (`decode_ucn`, `src/mccpp.c`, checked right after the hex value is parsed — both of its callers are identifier contexts). Verified vs gcc+clang: `int A;` (A) and `int \U0000d800x;` (surrogate) both rejected by all three; legal `int é;` (é) still compiles+runs and `$` (`$`) is correctly *exempted* by the range check. Tested via cli `ucn_identifier_range`; fires identically on all 5 arches (arm64/riscv64 spot-checked). NOTE (unchanged, deliberate): the < 00A0 rule is **not** applied to string/char literals — `"A"` stays accepted because **gcc also accepts it there silently** (only clang rejects), so mcc matches gcc on the lenient case. (Separately, a bare `$` in an identifier without a UCN is still unsupported — a pre-existing limitation orthogonal to this rule.)

## 3. Concepts, conversions & declaration specifiers (§6.2, §6.3, §6.7.1–§6.7.5)

### Atomic types & the `_Atomic` specifier/qualifier (§6.2.5, §6.7.2.4, §6.7.3)
- [~] **[BUG] §6.2.5p27 / §6.7.2.4 `_Atomic` has no atomic semantics** — DONE for scalar-integer read-modify-write (the silent-miscompile case): `_Atomic` now has its own type-flag bit `VT_ATOMIC_BIT` (0x20000) **in addition to** `VT_VOLATILE` (so all existing no-reorder handling still applies); a new `VT_QUALIFY` mask carries it through every qualifier strip/propagate site (incl. the i386/x86_64 backend `load`/`store`). In-language RMW on an `_Atomic` integer scalar (`a++`, `--a`, `a+=x`, `a-=x`, `a&=x`, `a|=x`, `a^=x`) is now lowered to the seq-cst `__atomic_<op>_fetch_<size>` / `__atomic_fetch_<op>_<size>` helpers (`gen_atomic_rmw`/`atomic_rmw_size` in `src/mccgen.c`, wired into `expr_eq` and `inc`). **Verified truly atomic**: 8 threads × 100k `counter++` yields exactly 800000 (was racy before); single-threaded results match gcc bit-for-bit incl. 8-bit wrap and 64-bit. Compiles on all 5 arches. ALSO DONE: **atomic pointer** `+= -= ++ --` (`gen_atomic_rmw` scales the integer rhs by the pointee size, then uses `__atomic_fetch_add/sub_<ptrsize>`) — verified vs gcc for `int*`/`char*`/`double*` pointers on all 5 arches. ALSO DONE: **integer `*= /= %= <<= >>=`** (the operators with no direct fetch helper) now lower to a **compare-exchange retry loop** (`gen_atomic_cas_rmw`/`atomic_cas_size`, mirroring `do { new = old OP rhs; } while(!__atomic_compare_exchange_N(...))` using private local slots + a `gind`/`gvtst`/`gsym_addr` backward loop). Verified vs gcc (`int`/`unsigned`/`short`/`long long`, all five ops), clean under `-Werror`, on all 5 arches; the contended exec test runs an identity CAS multiply across 8 threads to confirm the retry loop neither corrupts nor hangs. ALSO DONE: **float** atomic RMW (`_Atomic float`/`double`, any op) — routed through the same CAS loop, with the `desired` value read from its slot as an **exact-size integer** so it is passed in a GP register (the helper ABI), which also bit-reinterprets the float correctly. Verified vs gcc (`++`/`*=`/`+=`/`/=`/`-=` on float and double) on all 5 arches. **In-language scalar atomic RMW is now complete for every integer/float/pointer type and operator.** Tested via exec `atomic_inlang_rmw` + cli `atomic_rmw_unsupported`. ALSO DONE: **plain store** to a wider-than-word `_Atomic` scalar (e.g. `_Atomic long long g; g = v;` on a 32-bit target, which previously compiled to two non-atomic word writes — confirmed via i386 objdump) now lowers to `__atomic_store_<size>` (`gen_atomic_store_scalar`/`atomic_store_needs_libcall`, hooked in `expr_eq`'s `=` case; only fires when `size > PTR_SIZE`, so 64-bit targets and ≤word scalars keep their single-instruction store). Runtime-verified on i386 and arm under qemu. The atomics conformance test (`tests/qemu/conformance/atomics.c`) now also covers the plain 8-byte store and passes on all 5 arches × glibc+musl. ALSO DONE: the in-language plain *read* of a wider-than-word `_Atomic` integer scalar (`y = g;`, `g + 1`, `return g`, `foo(g)`, `if (g)`) — previously two non-atomic word loads on a 32-bit target. The hook is in `gen_cast` (`gen_atomic_load_scalar`, guarded by `atomic_lowering` so it doesn't re-fire inside the RMW/CAS/store/load helpers): every lvalue-to-rvalue conversion of such an object passes through `gen_cast` *before* the `_Atomic` qualifier is stripped, so it lowers to `__atomic_load_<size>`. Crucially the predicate (`atomic_store_needs_libcall`) excludes **float/double** — an aligned 8-byte FP load/store is already single-instruction atomic (`fldl`/`vldr`), and routing it through the integer-register helper would be an ABI mismatch. **Proven atomic** by a multithreaded torn-read detector on i386 (a writer alternating all-0/all-1 bits, reader sees no torn value). i386 `return g` now emits `call __atomic_load_8`; `double` reads stay single-instruction. Verified on i386/arm under qemu; native 24/24 and conformance 70/70 (incl. the in-language read in `atomics.c`) green. ALSO DONE: in-language plain store to an _Atomic **aggregate or >8-byte object** (`_Atomic struct g; g = v;`, `_Atomic long double ld; ld = x;`) — previously a non-atomic struct/multi-word copy — now lowers to the generic `__atomic_store(size,&dest,&val,order)` libcall (`gen_atomic_store_aggregate`/`atomic_store_needs_generic`, hooked in `expr_eq`; needs `-latomic`). Verified vs gcc (24-byte struct + long double → identical), references `__atomic_store` on all 5 arches; cli `atomic_inlang_aggregate_store`. RESOLVED N/A: **>8-byte fetch-ops** (`atomic_fetch_add` etc.) — mcc has no `>`word integer type (`__int128` unsupported) and fetch-ops on `float`/`long double` are invalid C (both mcc and gcc reject "integral or integer-sized pointer target type expected"), so there is nothing reachable to lower. ALSO DONE: in-language *read* of an _Atomic **aggregate / >8-byte object** (`Big r = g;`, `r = g;`, `*p = g;`) — the struct copy is taken from an atomic snapshot loaded via the generic `__atomic_load(size,&src,&tmp,order)` (`gen_atomic_load_aggregate`, hooked at the top of `vstore`, `atomic_lowering`-guarded). Verified vs gcc (`Big r1=g; r2=g;` → identical `789 789`), references `__atomic_load` on all 5 arches; conformance 70/70 (vstore hot-path regression-checked). cli `atomic_inlang_aggregate` covers store+load. **In-language atomic load/store is now complete for scalars, pointers, and aggregates.** RMW on `long double`/`>`8-byte errors cleanly. NOTE on "wider-than-word atomic fetch-ops": **C11 has no integer type wider than `long long`** — §6.2.5 standard integer types are `char/short/int/long/long long`; extended integer types are optional and `intmax_t` = `long long` (64-bit) is fully conforming. So the complete C11 integer-atomic fetch-op surface is `char`…`long long`, and `tests/qemu/conformance/atomics.c` now exercises `atomic_fetch_{add,or,and,xor,sub}` across **all four widths (8/16/32/64-bit)** — verified vs gcc and on all 5 arches × glibc+musl. `__int128` (a GCC extension, **not** C11) and `_Decimal128` (C23 / TR 24732, **not** C11) are out of the C11 scope and intentionally absent; their absence is C11-conformant.

DONE for the explicit `_Atomic(type-name)` form: `_Atomic(int[3])` and `_Atomic(int(void))` now error (`src/mccgen.c` after `parse_expr_type` in the `TOK__Atomic` case). A deliberately-broken build returns its failure code, confirming the program logic really executes. What this verifies is mcc's **code generation** on Mach-O across the full call surface — atomics, struct ABI, varargs (incl. the `snprintf` varargs call), and libc-call ABI. (4) **Mach-O against Apple's REAL libc** — `tests/qemu/run_macho_apple_libc.sh` (CMake test `macho-apple-libc`): two **verbatim** vendored Apple source sets are compiled by mcc for x86_64 Darwin, linked TOGETHER into Mach-O images with only a freestanding entry (no hand-written libc), and executed via the in-repo loader. (a) `tests/qemu/apple-libc/src/*.c` — Apple `Libc` `string/FreeBSD/` (`strcspn`/`strpbrk`/`strsep`/`memmem`/`strchrnul`/`strnstr`). (b) `tests/qemu/apple-libc/src-libplatform/*.c` — Apple `libplatform` `src/string/generic/`, the **core string/memory routines** `strlen`/`strcmp`/`strncmp`/`strcpy`/`strncpy`/`strlcpy`/`strlcat`/`strchr`/`strstr`/`strnlen`/`memmove`/`memcpy`/`memcmp`/`memchr`/`memccpy`/`bzero`/`memset`/`memset_pattern4`. These last are exactly the functions that ship as Darwin **commpage assembly** on macOS — but that asm is only an *optimization variant*; Apple ships this portable C as the functional equivalent (gated by `_PLATFORM_OPTIMIZED_*`, which `<platform/string.h>` defaults to 0), and it **runs fine as a Mach-O image off-Darwin**. Because the two sets are linked together, the FreeBSD funcs resolve `strlen`/`memcmp`/`memchr`/`strncmp` against Apple's real `libplatform` code, so **every** string/memory function under test is Apple's own source — no stand-in. (c) `tests/qemu/apple-libc/src-simple/string_io.c` — Apple `libplatform`'s **self-contained formatted-output engine** `__simple_bprintf`/`_simple_snprintf` (the locale-free printf Apple uses in low-level code like libmalloc's error reporting). Its fixed-buffer entry writes into the caller's buffer and never touches Mach VM, so Apple's genuine printf engine (`%d %i %u %o %x %X %p %c %s` + width/zero-pad/`l`/`ll`) runs as a Mach-O image; `apple_simple_conf.c` checks output byte-for-byte. Three self-checking programs (`apple_string_conf.c`, `apple_libplatform_conf.c`, `apple_simple_conf.c`) exercise the surface and exit 0. The compat shim (`shim-include/`) only supplies build glue (`__FBSDID`→nothing, `u_long`/`LONG_BIT 64`/`uintptr_t`/`INT_MAX`, no-op `libc_hooks_*`, FreeBSD `__weak_reference` alias, `<platform/string.h>` with `_PLATFORM_OPTIMIZED_* 0`, stubbed-but-never-called Mach-VM/`write`/`errno`); no source body is modified. (Debugging the `_simple` engine surfaced that an un-prototyped `strchr` makes the compiler truncate its pointer return to `int` — standard C UB for implicit declarations, not an mcc bug; fixed by prototyping it in the shim.) NOTE: the part of `libSystem` that remains genuinely unrunnable off-macOS is the **kernel-fused** remainder, each verified by reading the actual Apple source: `malloc` (`libmalloc` — magazine/nano/xzone zones on `os_unfair_lock`=kernel `__ulock`, Darwin TSD, `mach_vm_allocate`; NO portable C variant exists, unlike the string layer); FILE-backed/locale `stdio` (`Libc/stdio/.../vsnprintf.c` is bound to `xlocale`/`__v2printf`/`__sFILE`+`gdtoa` — reconstructing it would be re-authoring Apple's private subsystem, not build glue); and `dyld`/`pthread`/GCD/ObjC/Mach IPC (kernel/dyld-bound by definition). Those need a macOS/darling host; the boundary is documented in `tests/qemu/apple-libc/PROVENANCE.md`. This is the standard cross-compilation division: mcc (the compiler) emits the calls + correct Mach-O container/ABI; the hosted §7 library is the platform's — glibc/musl on ELF (80 checks), msvcrt on PE (16), and now **the entire core string/memory layer plus Apple's genuine self-contained printf engine of real libSystem on Mach-O**. All CMake tests self-skip when their tooling/cross compilers are absent. - [x] **[BUG] 32-bit ARM: 8-byte locals (`long long`/`double`/`_Atomic`) under-aligned** — found via the qemu atomics runtime test: the ARM prolog saved `{fp, ip, lr}` (12 bytes), so after `mov fp, sp` the frame pointer sat at an `8k+4` address; 8-byte locals (addressed fp-relative) landed at 4-byte alignment, violating AAPCS and **trapping 8-byte atomics** (`LDREXD`/`STREXD` require 8-byte alignment → `SIGBUS`).
